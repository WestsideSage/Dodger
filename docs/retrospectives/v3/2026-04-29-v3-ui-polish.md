# V3 UI/UX Polish & Performance Audit

**Date:** 2026-04-29
**Phase:** Pre-V4 Preparation
**Role:** Lead Front-End UX Engineer — *Codename: Sonnet*
**Surfaces Audited:** React Web Frontend (`frontend/`) · Tkinter Manager Mode (`manager_gui.py`)

---

## Project Trajectory

### WHERE WE WERE (V2)

The V2 UI was entirely Tkinter. It had no design system — colors, padding, and font choices were hardcoded at each call site. The interface felt more like a developer's debug sandbox than a sports management game. Raw player UUIDs appeared in reports, the match replay was nonexistent, and the off-season ceremony was a linear wall of text. There was no web surface at all; everything ran through the local Tkinter window.

### WHERE WE ARE (V3)

V3 delivered two parallel wins:

1. **Tkinter design foundation.** `ui_style.py` centralizes design tokens (`DM_PAPER`, `DM_BRICK`, `FONT_BODY`) and `ui_components.py` introduces reusable primitives — `StatCard`, `MetricStrip`, `RatingBar`, `UncertaintyBar`. The Hub layout, Roster separation, and Replay controls now have genuine visual hierarchy. Names appear instead of IDs in primary surfaces.

2. **React web foundation (V4 seed).** `frontend/` is a Vite + React + TypeScript SPA using the same "Golden Lap-inspired" retro design tokens from the UI Guide. `server.py` (FastAPI) exposes `/api/status`, `/api/roster`, `/api/tactics`, and `/api/sim/week`. Three screens exist: Hub, Roster, Tactics.

Both surfaces share the same `dodgeball_sim.db` and the same deterministic Python engine. The Tkinter app remains the primary playable client; the React app is a working foundation at ~20% feature parity.

The main liability on the Tkinter side is scale: `manager_gui.py` has grown to **4,145 lines** with all state, data-fetching, and view construction entangled. On the React side, the three screens are functional but shallow — raw UUIDs in key display fields, invisible player traits, no feedback on saves, and missing the five V3 pacing controls.

### WHERE WE ARE GOING (V4)

V4's primary goal is **web feature parity** with the V3 Tkinter app. The engineering priorities in UX order:

1. Replace raw UUID display with resolved names across all web screens.
2. Port the five V3 pacing controls to the Hub (Play Next Match, Sim Week, Sim To Next User Match, Sim Multiple Weeks, Sim To Milestone).
3. Add League Context screens: Standings, Schedule, News Wire.
4. Add Scouting and Recruitment views.
5. Port the Match Day Replay to an HTML5 `<canvas>` component driven by `GET /api/match/{id}/replay`.
6. On the Tkinter side: decompose `manager_gui.py` into a `screens/` directory as a pre-condition for any further Tkinter feature work.

## React Component Debt

The following issues are grounded in direct inspection of the source files.

### `App.tsx` — Tab Navigation Shell

**Debt 1 — No URL routing.** `activeTab` is React state only. Navigating to `/?tab=roster` does nothing; refreshing the page always lands on Hub. As V4 adds 6+ screens, this becomes a serious DX and UX problem. Fix: `URLSearchParams` (zero-dependency) or React Router.

**Debt 2 — Tab label display.** Tabs render `{tab}` directly — all lowercase (`hub`, `roster`, `tactics`). The display names should be mapped: `{ hub: 'Hub', roster: 'Roster', tactics: 'Coach Policy' }`. The Tactics screen's own `<h2>` already says "Coach Policy" but the tab says "tactics".

**Debt 3 — No keyboard shortcuts.** Power users have no way to jump between screens without mouse clicks.

### `Hub.tsx` — Manager Hub

**Debt 4 — Duplicated fetch logic.** The component has two separate implementations of the same `/api/status` call: the `useEffect` at mount (with a `cancelled` flag) and `refreshStatus()` (without one). These should be unified into a single `useCallback`-memoized function with a proper `AbortController`.

```tsx
// Current: two paths for the same fetch
useEffect(() => { fetch('/api/status').then(...) }, []);
const refreshStatus = () => fetch('/api/status').then(...);
```

**Debt 5 — Raw UUID display.** Three fields show machine identifiers instead of human labels:
- `player_club_id` → shows a UUID; should show the club's display name
- `season_id.split('-')[0]` → shows the first UUID fragment; should show "Season 1"
- `match_id.substring(0, 8)` → shows a UUID prefix; should show "Wk 3: Aurora FC vs Redline"

**Debt 6 — Machine state key as phase label.** `status.state.state.replace('_', ' ')` surfaces internal values like `"season active in match"` verbatim. V4 needs a display-name map aligned with the `CareerStateCursor` state machine.

**Debt 7 — Missing Hub context.** The V3 Tkinter Hub shows: next opponent, current W/L record, standings position, and last match result. The web Hub shows none of these. The simulation button lacks any context about what will be simulated.

**Debt 8 — Single pacing action.** V3 Tkinter exposes 5 pacing controls; the web Hub exposes 1. This is the largest functional gap on the Hub screen.

### `Roster.tsx` — Team Roster Table

**Debt 9 — O(n²) starter-badge check.** Every row calls `data.default_lineup.includes(player.id)` during render. With a 15-player roster this is 225 string comparisons per render cycle.

```tsx
// Current (O(n) per row):
data.default_lineup.includes(player.id)

// Fix (O(1) per row, computed once):
const starterSet = useMemo(() => new Set(data.default_lineup), [data.default_lineup]);
starterSet.has(player.id)
```

**Debt 10 — `PlayerTraits` entirely invisible.** `types.ts` defines `potential`, `growth_curve`, `consistency`, and `pressure`. None appear in the Roster table. This is the single most important missing data dimension — potential/ceiling is what scouting and recruitment are about. At minimum, a `Potential` column should be visible.

**Debt 11 — No OVR composite.** The five rating bars (Accuracy, Power, Dodge, Catch, Stamina) give a useful breakdown but there is no quick-scan composite score to sort by. Adding a computed `OVR` column would make roster decisions dramatically faster.

**Debt 12 — No column sorting.** Clicking a column header does nothing. Sorting by Dodge or Age is a core roster management action.

**Debt 13 — Bars-only rating display.** `RatingBar` only shows a numeric value when the `label` prop is passed. In the Roster table, `label` is never passed — ratings are bars with no numbers. A user hovering over a bar cannot read the exact value.

**Debt 14 — No sticky header.** Long rosters (10–15 players) require scrolling. Column headers scroll off-screen with no sticky positioning.

### `Tactics.tsx` — Coach Policy

**Debt 15 — Silent save.** After clicking "Save Tactics," nothing visible happens. The success branch is a `// Optionally show success message` TODO comment. This is the highest-priority single-line UX fix in the entire frontend.

**Debt 16 — Always-enabled save button.** The "Save Tactics" button is enabled on page load before any change is made. A `isDirty` state flag (compare current policy to a `savedPolicy` ref) would prevent phantom saves.

**Debt 17 — No reset to defaults.** There is no way to restore coach policy to its original/recommended values without knowing what those are.

**Debt 18 — Related sliders not visually grouped.** `rush_frequency` and `rush_proximity` are semantically one "rushing system" but are separated into adjacent grid cells with no visual indication of their relationship. A light group box or a divider label ("Rush Behavior") would add clarity.

**Debt 19 — Full object spread on every drag event.** `updatePolicy` does `{ ...prev, [key]: value }` on every pixel of slider movement. At 8 fields this is fine; as the domain expands toward 15+ tendency fields (a likely V4 scenario), this pattern accumulates garbage. A `useReducer` with `{ type: 'SET_FIELD', key, value }` is the correct long-term shape.

### `ui.tsx` — Design System Components

**Debt 20 — `RatingBar` hardcoded thresholds.** The 80/60/40/20 breakpoints are always correct for a 0–100 overall scale, but may be wrong for per-stat interpretation (a 60 Stamina is "good" while a 60 Accuracy might be "average" for a thrower). An optional `thresholds` prop would allow caller-controlled color breakpoints.

**Debt 21 — No color legend.** The five-tier color system (teal → sage → mustard → orange → danger) is used across the app but never explained. A `<RatingLegend />` chip or tooltip on the Roster header would surface this.

**Debt 22 — `TendencySlider` inline `accentColor`.** The slider thumb color is set with `style={{ accentColor: 'var(--color-brick)' }}` — a raw inline style. Every other interactive element uses Tailwind utilities. This inconsistency breaks the single-source-of-truth principle for the design token system.

**Debt 23 — Missing ARIA labels on sliders.** Native `<input type="range">` is keyboard accessible, but has no `aria-label` or `aria-valuetext`. Screen readers will announce the numeric value but not what it means. `aria-label={label}` should be added at minimum.

## Tkinter Component Debt

*(Retained and refined from prior audit pass.)*

1. **`manager_gui.py` monolith (4,145 lines).** All application state, data-fetching, view construction, and event wiring are in a single file. Any new V4 screen addition makes an already-unmanageable file larger. Priority: decompose into a `screens/` directory (`HubScreen`, `RosterScreen`, `MatchDayScreen`, `ScoutingScreen`).

2. **Treeview boilerplate repetition.** Roster, Standings, Free Agents, and Draft Prospects all independently set up `ttk.Treeview` with near-identical column-definition and row-insertion patterns. A `DataTable(frame, schema, data)` helper would absorb this.

3. **Pacing & navigation rail entanglement.** The left-hand navigation and pacing controls (Play Next Match, Sim Week, etc.) are constructed and wired inside the main application class rather than in an isolated `GlobalNav` component. This makes the nav impossible to test or reuse.

4. **Scattered `PlayerInspector` logic.** Displaying a player's attributes, traits, and `UncertaintyBar` happens inline across Roster, Scouting, and Recruitment screens with no shared abstraction.

## Layout Consistency Issues (React)

| Issue | Details |
|-------|---------|
| **Mismatched max-widths** | `Hub` → `max-w-4xl`, `Tactics` → `max-w-3xl`, `Roster` → none. All three screens should use `max-w-5xl` to match the App shell. |
| **Error state theme break** | Error divs use `bg-red-50` — a default Tailwind class not in the design token system. Should be `bg-[var(--color-cream)]` with `border-[var(--color-danger)]`. |
| **Plain-text loading states** | All three screens show "Loading hub...", "Loading roster...", "Loading tactics..." as unstyled text. A `<LoadingState label="..." />` skeleton component would be consistent and more professional. |
| **No Tactics save feedback** | The save flow has no acknowledged state. The button should momentarily show "Saved ✓" or a `StatChip` confirmation. |
| **Tab label/screen title mismatch** | Tab says `tactics`; screen `<h2>` says "Coach Policy". These must agree. |

## Rendering Performance

### React

| Component | Issue | Severity | Recommended Fix |
|-----------|-------|----------|-----------------|
| `Roster` | `default_lineup.includes(id)` is O(n) per row → O(n²) total per render | **Medium** (15 players fine; 50-row scouting board is not) | `useMemo(() => new Set(default_lineup), [default_lineup])` |
| `Tactics` | Full policy object spread `{ ...prev, [key]: v }` on every slider drag event | **Low** (8 fields) → **Medium** (15+ fields) | `useReducer` with `SET_FIELD` action |
| `Tactics` | Slider `onChange` fires on every pixel movement — no debounce | **Low** (no auto-save today; will matter for future auto-save) | 50ms debounce via `useCallback` + `setTimeout` |
| `Hub` | `refreshStatus` is re-created on every render — no `useCallback` | **Low** (not a prop today) | `useCallback` preemptively |
| `RatingBar` | No `React.memo` — re-renders on any parent state change | **Low** (15 rows) → **Medium** (scouting board, 50+ rows) | `export const RatingBar = React.memo(...)` |

### Tkinter

| Pattern | Issue | Fix |
|---------|-------|-----|
| Destructive redraws | `frame.destroy()` + full repopulate on state update | `tk.StringVar` / `itemconfig()` mutation |
| Treeview inserts | Row-by-row insertion blocks the UI thread | Batched insert inside `after_idle` |
| `UncertaintyBar` canvas | Full `delete("all")` + redraw on every update | Update existing canvas item handles with `coords()` / `itemconfig()` |
| `CourtRenderer` | Full court background redrawn each replay tick | Render static background once to a separate layer; redraw only player tokens |
