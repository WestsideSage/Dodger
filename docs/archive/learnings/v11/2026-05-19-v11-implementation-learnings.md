# V11 USAD Rules Integration - Implementation Learnings
Date: 2026-05-19

## Key Architectural Decisions

### 1. Translator-Layer Integration vs. Duck-Typing
We chose to implement a dedicated translation layer (`OfficialEvent` to generic `MatchEvent`) rather than using duck-typing across the engine and event buses.
- **Rationale**: 
  - **Preserves Invariants**: Keeps the core invariant that events generated by the match engines are canonical and structurally simple.
  - **Pipeline Isolation**: The statistics compilation (`official_stats.py`), Match Replay, and database persistence layers remain completely agnostic to the deep complexities of USA Dodgeball-specific rulesets.
  - **Decoupled Evolution**: Allows the ruleset definitions to evolve independently without forcing changes onto the downstream generic subsystems.

### 2. Career-Creation-Only Opt-In
Ruleset selections (Generic vs. USAD Foam/No-Sting/Cloth) are strictly evaluated and locked at career creation.
- **Rationale**:
  - **Zero Migration Risk**: Guarantees that existing V1–V10 saves remain untouched and stable on the generic engine.
  - **Data Integrity**: Avoids complex structural conversions mid-career, which would disrupt standings, league memory, and career stat histories.

### 3. Scripted vs. Autonomous Engine Split (Phase 8A vs. Phase 8C+)
Developing a scripted engine harness in addition to the autonomous tick loop was an essential verification strategy.
- **Rationale**:
  - **Controlled Test Surface**: The scripted engine consumes predetermined actions, enabling fast, isolated, deterministic assertions of boundary cases (e.g., specific ball state sequences, catch queue configurations, or clock advance ticks) without fighting chaotic AI random selection.
  - **Contract Solidification**: Allowed us to define and freeze the replay contract schemas in `replay_contracts.py` before finalizing complex behavioral tactics.

### 4. Shipping Warnings + Blue Cards Only
We restricted the V11 discipline model to verbal warnings and blue cards, leaving yellow and red cards as placeholders.
- **Rationale**:
  - **Persistence Boundaries**: Yellow and red cards have long-running multi-match or tournament-suspension implications.
  - **Scope Control**: Implementing full suspension states would require schema migrations to persist banned players across career weeks, which violates our zero-migration stability contract.

## Systems & Environments Insights

### 1. Uvicorn Port-Collision & Stale Code Symptoms
During Playwright browser testing and local development, we observed intermittent "stale code" symptoms where new backend responses didn't match the active code changes.
- **Cause**: Uvicorn server processes sometimes survived test termination, creating orphan processes that continued holding port `8000`. Subsequent runs launched a new server that could not bind to the port, causing the test runner to communicate with the stale orphan process.
- **Resolution**: Proactively checking and terminating orphan python processes holding port `8000` is essential during test suite runs. Playwright's `webServer.reuseExistingServer: !process.env.CI` was carefully verified to ensure proper setup.
