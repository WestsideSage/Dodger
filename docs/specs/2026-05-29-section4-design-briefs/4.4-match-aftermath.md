# Brief 4.4 — Match Aftermath Hierarchy

**Phase 8 Claude Design handoff. Logic settled in Phases 1–7.**

---

## 1. Teardown

**Component:** `frontend/src/components/MatchWeek.tsx` — `renderPostSimMode`, the post-simulation state.
**Key child components:** `Headline`, `MatchScoreHero`, `PrimaryFactorCard`, `TacticalSummaryCard`, `KeyPlayersPanel`, `FalloutGrid`, `ReplayTimeline`, `NextBestImprovementPanel`, `AftermathActionBar`.
**Backing type:** `Aftermath` in `frontend/src/types.ts`.
**Score rendering canonical module:** `frontend/src/components/match-week/matchResult.ts` → `formatScoreline`.

**What exists today:**

The aftermath reveals in four timed stages (0–4), each adding content. Stage 0 = Headline. Stage 1 = MatchScoreHero + verdict + PrimaryFactorCard + body paragraphs. Stage 2 = TacticalSummaryCard + KeyPlayersPanel. Stage 3 = FalloutGrid + ReplayTimeline + NextBestImprovementPanel. Stage 4 = AftermathActionBar or EliminationCeremony.

**Score hero (critical — read this before designing):** `MatchScoreHero` calls `formatScoreline` from `matchResult.ts`. For an official-foam career (`scoring_model !== 'legacy'`), `isOfficial = true` and the hero number displayed is **game points** (`home_game_points` / `away_game_points`), not survivors. Survivors appear in the supporting detail line below the hero number as "N survivors (Final)". For legacy careers, survivors are the hero number. The center label reads "Final (USAD OFFICIAL_FOAM)" for official, "Final" for legacy.

**What is silent / redundant today:**

- The timed reveal (stages 0–4) staggered over 1-second intervals prevents the player from seeing the score until stage 1. A space-bar or click skips to stage 4. This is a UX affordance, not a layout problem — but briefs should note it so the designer does not break the `revealStage` system.
- The `verdict` string (server-authored verdict) renders in the same visual weight as the `PrimaryFactorCard` but without the structured confidence chip. On a decisive win it duplicates the headline. If the brief's hierarchy promotes `primary_factor` as primary explanation, the `verdict` becomes redundant (secondary at best).
- The `body` paragraphs (audience-tagged YOU/THEM/RESULT) carry the tactical narrative. They are rendered with colored left-border badges but no spatial grouping — all paragraphs are in a single `gap`-spaced grid. The YOU/THEM/RESULT distinctions collapse visually into a list of similarly-weighted items.
- The `TacticalSummaryCard` and `KeyPlayersPanel` share a `.command-analysis-row` flex row. On a 390px viewport they stack. The label "Tactical Summary" vs "Key Players" is not differentiated visually — both are `dm-panel` cards in the same weight.
- `NextBestImprovementPanel` is positioned after the `FalloutGrid` and `ReplayTimeline` — it is only present on a loss. This positioning is correct (the player is still digesting the result), but the panel currently uses the same visual weight as the training/standings fallout cards, diluting its signal.
- The `ReplayTimeline` is collapsed by default. On a decisive result the `PrimaryFactorCard` already tells the story — the timeline is the drill-down, not the headline.

**Phase 1–7 changes already settled:**
- **Set scoring (Phase 4b):** new careers use `scoring_model = 'official_foam'`; `home/away_game_points` are present and are the hero numbers via `formatScoreline`. Survivors are demoted to the sub-detail. The design must reflect this: the large animated number is game points for official careers.
- **PRIMARY FACTOR (Phase 4b):** `derive_match_explanation` now uses `point_margin` from the official game-point gap. A 0-4 loss and a 6-0 win are never "inconclusive." The `primary_factor.confidence` can be `high`, `medium`, or `low`; the corresponding chips are "HIGH CONFIDENCE", "LIKELY", "INCONCLUSIVE". Do not redesign the confidence vocabulary — it is settled.
- **Moment events (Phase 4a/4b):** `moment_events` now flow from the official engine through `OfficialEngineAdapter` into the `match_end` context, which `aftermath_context.moment_events_from_payload` consumes. The `ReplayTimeline` renders `dramatic_catch`, `gassed_collapse`, `flood_throw` as inline beats; `late_game_escape` as a `LateGameBanner`; `one_v_one_finale` as a `OneVOneBanner`; `comeback` as a `ComebackCard`. These are all live for official-foam careers.

## 2. Information Hierarchy

**Primary:** The result and the honest explanation — `MatchScoreHero` (game points for official, survivors for legacy) + `PrimaryFactorCard` (the single deciding factor with its confidence level). Together these answer "what happened and why."

**Secondary:** The tactical and performance evidence — `TacticalSummaryCard` (turning point + evidence lanes), `KeyPlayersPanel` (top performers), the audience-tagged `body` paragraphs. These answer "how did this unfold."

**Tertiary / support:** `FalloutGrid` (week fallout: who grew, standings shift, prospect pulse), `ReplayTimeline` (the drill-down event log with moments), `NextBestImprovementPanel` (loss-only actionable next steps).

**Cut or demote:**
- The `verdict` string — if `primary_factor` is present, the verdict is redundant. Consider rendering verdict only as a fallback when `primary_factor` is absent (which should be rare post-Phase 4b). Do not render both at full weight.
- The individual `body` paragraph audience badges (YOU/THEM/RESULT) should guide layout/grouping, not appear as prominent chips on every line. Consider grouping YOU paragraphs, THEM paragraphs, and RESULT paragraphs into distinct sub-sections rather than interleaving them.

## 3. Data Inventory

**From `Aftermath` type:**
- `headline`: string
- `match_card`: `{ home_club_id, away_club_id, winner_club_id, home_survivors, away_survivors, scoring_model?, home_game_points?, away_game_points? }` or null
- `verdict`: optional string
- `primary_factor`: optional `{ code, title, sentence, confidence: 'high'|'medium'|'low', evidence_chips: string[] }`
- `body`: `AftermathParagraph[]` — each has `{ text, audience: 'you'|'them'|'result' }`
- `top_performers`: optional `TopPerformer[]` — `{ player_id, player_name, club_name?, score, eliminations_by_throw, catches_made, dodges_successful }`
- `player_growth_deltas`: `Array<{ player_id, player_name, attribute, delta }>`
- `development_feedback`: optional `{ focus, focus_label, summary, progress, players: string[] }`
- `standings_shift`: `Array<{ club_id, club_name, old_rank, new_rank }>`
- `recruit_reactions`: `Array<{ prospect_id, prospect_name, interest_delta, evidence }>`
- `narrative_beats`: optional `{ was_shutout, largest_deficit, lead_changes, selected_plan_label, actual_plan_executed }`
- `improvement_panel`: optional (loss-only) `Array<{ category: 'position_group'|'condition'|'recruit', title, detail }>`
- `playoff_resolution`: optional `{ decided_by, narrative_note, winner_club_id, loser_club_id, stage, player_outcome }`
- `elimination`: optional (loss eliminates player from playoffs)
- `championship`: optional (player wins the title)
- `bye_recovery`: absent on match weeks (present only on bye)

**From `MatchReplayResponse`** (loaded separately via `commandApi.replay(matchId)`):
- `report.turning_point`: string
- `report.evidence_lanes`: `CommandDashboardLane[]` — `{ title, summary, items: string[] }`
- `report.top_performers`: `TopPerformer[]`
- `moment_events`: `MomentEvent[]` — up to 4 kinds (dramatic_catch, late_game_escape, one_v_one_finale, comeback) for official careers

**From `CommandCenterSimResponse`:**
- `dashboard.week`: integer
- `dashboard.stage`: string ("Regular Season" or "Semifinals" / "Final" for playoffs)
- `dashboard.result`: "Win" / "Loss" / "Draw"
- `dashboard.lanes`: `CommandDashboardLane[]` (fallback if replay not loaded)

**Score rendering (canonical):** `formatScoreline(match_card)` from `matchResult.ts`:
- Returns `{ isOfficial, centerLabel, home: { value, survivors }, away: { value, survivors } }`
- `value` = game points when `isOfficial`, survivors otherwise
- The big animated number in `MatchScoreHero` is `value`, not always survivors

## 4. Constraints

- Desktop-first matrix (1440x900 / 1366x768 / 1280x720). Mobile layouts are explicitly a non-goal. The `.command-analysis-row` currently uses flexbox — on narrow viewports it stacks, which is fine, but do not optimize for 390px.
- The timed reveal (`revealStage` 0–4) must be preserved. The designer can reorganize content within each stage but should not collapse stages (they serve the "building tension" reveal loop). Stage 0 = Headline, Stage 1 = Score + verdict/factor + body, Stage 2 = tactical + performers, Stage 3 = fallout + timeline + improvement, Stage 4 = CTA.
- Semantic markup: `MatchScoreHero` should carry `aria-label` that states the scoring model (already implemented: `aria-label="Final game score"` for official, `"Final survivor score"` for legacy — preserve).
- No new dependency, no new API endpoint. Replay loads via an existing `commandApi.replay(matchId)` call.
- "Explain, don't decide" — `primary_factor` is the engine's explanation; do not author alternative explanations. If `primary_factor` is absent, show the verdict string as the fallback explanation.
- The `NextBestImprovementPanel` is loss-only; do not render it on wins regardless of layout logic.

---

## 5. Success Criteria

1. A player can read the score and the primary explanation (what decided the match) without scrolling past a fold.
2. For official-foam careers, the hero number is game points (e.g., "3–1"), not survivors. Survivors appear as a supporting detail. The design must not invert this.
3. `PrimaryFactorCard` never shows "INCONCLUSIVE" confidence on a decisive set-score result (maintained from Phase 4b fix — do not regression-test this; confirm only that the card has the space to be prominent).
4. The YOU/THEM/RESULT body paragraphs are grouped or visually differentiated — not an undifferentiated flat list.
5. Moment event beats (dramatic_catch, late_game_escape, etc.) are accessible without expanding the timeline if they are the most dramatic element of the match. Consider surfacing at least one moment beat above the fold when present.
6. The `verdict` string does not render at full weight alongside `PrimaryFactorCard` on the same match — at most one should be primary at a time.
7. No horizontal overflow at 1280x720 minimum desktop width.
8. The `EliminationCeremony` and `ChampionshipHero` special-case screens are not affected — they are already distinct components that replace the normal aftermath for those events.
