# Brief 4.7 — Policy Editor Restyle

**Phase 8 Claude Design handoff. Logic settled in Phases 1–7.**

**The logic work for this screen is done. Phase 6 (D6) removed the redundant right-of-box label echo. This brief is the visual restyle only.**

---

## 1. Teardown

**Component:** `frontend/src/components/match-week/command-center/PolicyEditor.tsx`
**Backing type:** `CoachPolicy` in `frontend/src/types.ts`
**Voice register:** `useVoiceRegister(1)` — `t('policy.{key}.{value}.label')` / `t('policy.{key}.{value}.preview')`

**What exists today (post-Phase 6 de-dup):**

Five policy rows, grouped logically by an "Opening rush" sub-header that appears before the `rush_commit` row. Each row renders:
- A row label (`approach`, `target_focus`, etc.)
- A flex-wrap row of pill buttons — one per option value, styled as radio buttons (`role="radio"`, `aria-checked`). Selected pill: orange border + amber text on a slight orange background. Unselected pill: gray border + muted text on dark background.
- A `<p>` preview description (the `t('policy.key.value.preview')` string) that updates when the selection changes, with `aria-live="polite"`.

The redundant right-of-box label echo was removed in Phase 6 — the selected option is now conveyed only by the highlighted pill and described by the preview line. This is correct and settled.

A status badge in the header shows "Live" (cyan) or "Locked" (gray) to indicate whether edits are accepted. The `disabled` prop locks all pills and dims the badge.

Arrow-key navigation: `onKeyDown` on the `role="radiogroup"` div handles ArrowLeft/ArrowRight/Up/Down to cycle through options.

**What is still visually underdesigned:**

- The five rows are presented in an undifferentiated flat stack. The "Opening Rush" sub-group has a mono-font kicker label above `rush_commit`, but the visual weight of this grouping is weak — it looks like a stray line of copy, not a section header.
- The three core-policy rows (Approach, Target Focus, Catch Posture) and the two opening-rush rows (Commit, Target) are not visually distinguished as different phases of the match plan. A player new to the editor cannot tell at a glance that the bottom two rows govern only the opening rush.
- The pill buttons are the only interactive element but they look like toggle buttons, not radio buttons — there is no visual metaphor connecting "selecting Approach: aggressive" to "this is the play style for the entire match."
- The preview description text (`aria-live="polite"`) updates correctly but is rendered in identical small gray type for all five rows. The preview for Approach changes the overall match philosophy; the preview for Opening Rush Target changes a single moment at match start. These carry different decision weights but look the same.
- The `disabled` / "Locked" state changes only the badge text and dims the pills — there is no shaded overlay or structural signal that the entire editor is locked. A player might not notice the "Locked" badge and try to interact.
- No visual connection between the Policy Editor and the "Today's plan" heading — the heading says "Today's plan" but the copy registers the choices as five independent settings rather than a coherent plan.

**Phase 1–7 changes that are settled and must not be redesigned:**
- **De-duplication (Phase 6):** The right-of-box label echo of the selected option was removed. The three-showing-per-category problem is gone. Do not re-introduce any label echo.
- **CoachPolicy v2 (Plan C):** The five policy keys (`approach`, `target_focus`, `catch_posture`, `rush_commit`, `rush_target`) are the settled v2 model. Do not add or remove rows.
- **Aria semantics (Plan C):** `role="radiogroup"` / `role="radio"` / `aria-checked` / arrow-key navigation are already implemented and tested. The restyle must preserve these roles.
- **Voice register:** `t()` calls must be preserved — the design cannot replace voice-register labels with hardcoded strings.

## 2. Information Hierarchy

**Primary:** The selected state of each row — which option is active. The pill highlighting does this correctly; the restyle should make the active pill unmistakably selected at a glance without eye-hunting through five rows.

**Secondary:** The preview description — the consequence of the current selection. For high-stakes rows (Approach, Catch Posture), the preview is a decision aid. For lower-stakes rows (Opening Rush Target), it is context. The restyle may differentiate preview weight by row.

**Tertiary:** The Opening Rush sub-group framing — the player should understand that `rush_commit` and `rush_target` govern only the opening moment. The current mono-font kicker communicates this but weakly.

**Cut (do not re-introduce):**
- The right-of-box label echo of the selected option's text (removed in Phase 6).
- Any third representation of the selected value beyond the pill and the preview.

## 3. Data Inventory

**Policy fields available (`CoachPolicy`):**
- `approach`: `'aggressive' | 'patient' | 'mixed'`
- `target_focus`: `'their_stars' | 'ball_holders' | 'spread'`
- `catch_posture`: `'go_for_catches' | 'play_safe' | 'opportunistic'`
- `rush_commit`: `'all_in' | 'balanced' | 'hold_back'`
- `rush_target`: `'nearest' | 'strongest_side' | 'center'`

**Component props:**
- `policy: CoachPolicy` — current values
- `disabled?: boolean` — whether the editor is locked (post-sim)
- `error?: string | null` — server error string to display as `role="alert"`
- `onChange: (nextPolicy: CoachPolicy) => Promise<void> | void` — called on any change

**Row definition (from the `ROWS` constant in the component):**
```
{ key: 'approach', label: 'Approach', values: ['aggressive', 'patient', 'mixed'] }
{ key: 'target_focus', label: 'Target focus', values: ['their_stars', 'ball_holders', 'spread'] }
{ key: 'catch_posture', label: 'Catch posture', values: ['go_for_catches', 'play_safe', 'opportunistic'] }
{ key: 'rush_commit', label: 'Commit', values: ['all_in', 'balanced', 'hold_back'], group: 'opening-rush' }
{ key: 'rush_target', label: 'Target', values: ['nearest', 'strongest_side', 'center'], group: 'opening-rush' }
```

**Voice register keys for labels and previews:**
- `t('policy.{key}.{value}.label')` → rendered pill text
- `t('policy.{key}.{value}.preview')` → rendered description line

## 4. Constraints

- Desktop-first matrix (1440x900 / 1366x768 / 1280x720). Mobile layouts are explicitly a non-goal. Pills must wrap cleanly for desktop stress viewports.
- Aria semantics must be preserved: `role="radiogroup"` on the group div, `role="radio"` and `aria-checked` on each pill button, `aria-labelledby` pointing to the row label, `aria-live="polite"` on the preview paragraph.
- Arrow-key navigation (`ArrowLeft`/`Right`/`Up`/`Down` cycle through options) must remain functional — the event handler is on the `role="radiogroup"` div.
- `tabIndex={isSelected ? 0 : -1}` on pills implements roving tab index — this must be preserved.
- No new dependency. The `useVoiceRegister` hook and `t()` calls must remain.
- The `error` prop renders as `role="alert"` — preserve semantic placement.
- The `disabled` state must visually communicate that the editor is locked, not just dim the pills.

---

## 5. Success Criteria

1. A player can identify the selected option for each of the five rows in a single visual scan — no eye-hunting for the active state.
2. The Opening Rush sub-group (rush_commit + rush_target) is visually contained as a distinct section from the three core-policy rows.
3. The preview description for high-stakes rows (Approach, Catch Posture) carries more visual weight than for lower-stakes rows (Opening Rush Target). Alternatively, all preview lines may share weight if the grouping achieves the differentiation.
4. The `disabled` / "Locked" state communicates at a section level (the whole editor is locked), not only in a small header badge.
5. No redundant third representation of the selected option — pill + preview is the complete representation (maintained from Phase 6).
6. Arrow-key navigation still cycles through options (functional, not visual — but the restyle must not break the event handler structure).
7. No horizontal overflow at 1280x720 minimum desktop width.with three-option pill rows.
8. The error alert (`role="alert"`) remains visually prominent when present.
