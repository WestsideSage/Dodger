I have executed the Chaos Monkey QA breakdown on the Dodgeball Manager V3 local state and save system based on reverse-engineering the project's logic and architecture. Below are the findings for each edge case test:

### 1. Save Scumming (Offseason Ceremony Force Quit)
*   **Result (DB Corruption Risk): Low to None**
*   **Reproduction/Behavior**:
    Force-quitting the app (e.g., ALT+F4 or killing the process) directly during the offseason ceremony transitions (like `ratify_records` or `induct_hall_of_fame` passing cursor enums) does *not* result in a monolithic save file corruption. Save operations are committed incrementally through functions like `save_season()` or `save_recruitment_signings()` instead of a single fragile JSON/Binary file dump.
*   **Edge Case Discovered**: While the database won't corrupt, state desync *can* occur. If a user force-quits halfway through offseason updates, the `CareerState` cursor might remain stuck on `SEASON_COMPLETE_OFFSEASON_BEAT` but part of the stats or signing arrays may have already persisted.
*   **Recommendation**: Wrap the offseason ceremony database updates in a single SQLite transaction `BEGIN ... COMMIT` block.

### 2. Input Validation (Club Names & Custom Tactics)
*   **Result (String Processing / Exceptions): Handled, but rigidly typed**
*   **Reproduction/Behavior**:
    *   **Club Naming**: Injecting extreme special characters (`!!!Extreme&&^^Name???`) does not crash the UI. The internally mapped `_slugify_club_name(name)` function sanitizes inputs strictly using a regex `[^a-z0-9]+`, forcing the string to remain alphanumeric/lowercase. Injecting empty buffers (`""`) falls back defensively to naming the team "Expansion Club".
    *   **Custom Tactics Sliders**: Attempting to inject raw negative/extreme string numbers bypassing the `ttk.Progressbar` will trigger a raw `ValueError(f"{label} must be numeric.")` from the `CoachPolicy` handlers instead of failing cleanly, which could crash the GUI layer if directly caught by a Tkinter callback without a protective `try/except` block bounding it to percentages (0-100%).

### 3. Sequence Breaking (Navigating to Playoffs Early)
*   **Result (UI Routing state injection): Blocked by State Validations**
*   **Reproduction/Behavior**:
    Attempting to force the UI cursor to Playoffs mode before the regular season sets `SEASON_COMPLETE` fails gracefully. The UI relies on `_route_from_cursor()` checking `CareerState`. Furthermore, any manual injection attempting to call playoff modules checks `_playoffs_enabled()` against `load_season_format(self.conn, self.season_id) == PLAYOFF_FORMAT`. If the internal condition isn't met, the routing sequence refuses to repaint the UI to Playoffs view.

### 4. Roster Edge Cases (0-Man Roster on Match Day)
*   **Result (Match Engine Loop crashes): Defensive Trap Tripped**
*   **Reproduction/Behavior**:
    To reproduce, fire/release all players utilizing `_release_player_to_pool()` down to an empty roster and attempt to trigger the next week's match. A fatal infinite loop inside the dodgeball turn resolution *does not* occur. Instead:
    1. The GUI explicitly traps invalid roster deployments, raising a hard `ValueError("Each team must have at least one player.")` which acts as an unhandled UI exception blocking the user entirely.
    2. If forced functionally (e.g. bypassing GUI), `engine.py` defensively trips win constraints instantly due to the empty array `any(len(team.living_players()) == 0 for team in teams)`. 
*   **Recommendation**:
    To prevent the hard application crash when firing all players, add a validation inside the firing UI preventing the final player on a roster from being released, or allow the match simulation to automatically concede games (forfeit) if a roster hits 0 players to prevent unhandled validation crashes in the GUI thread.
