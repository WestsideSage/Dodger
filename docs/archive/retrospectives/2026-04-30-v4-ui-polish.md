# V4 UI Polish Retrospective

**Date:** 2026-04-30  
**Milestone:** V4 Web Architecture Foundation  
**Role:** Lead Front-End UX Engineer  
**Scope:** React component DX, dashboard UX, information density, and render-risk audit before V5 planning.

## Project Trajectory

### WHERE WE WERE

V3 delivered the experience rebuild and made Manager Mode feel more complete, but the UI was still constrained by legacy surfaces and first-pass web scaffolding. The player-facing product had useful screens, but agent inspection, browser verification, and frontend iteration were not yet the default workflow.

The main UI limitations entering V4 were:

- The web client existed as a foundation, not as the canonical dashboard experience.
- Dense manager information was present in the domain model, but not consistently shaped into scannable React tables.
- Match pacing and replay controls were useful, but still felt like parity controls rather than a polished match-day cockpit.
- Scouting and recruitment depth existed in the simulation, but not yet as a scalable React workflow.

### WHERE WE ARE

V4 moved Dodgeball Manager onto a Vite + React + TypeScript frontend backed by FastAPI. The shipped web surfaces are Hub, Roster, Tactics, Match Replay, Standings, Schedule, and News. That is the right product foundation for V5.

The component library is still very small. `frontend/src/components/ui.tsx` provides `Card`, `StatChip`, `RatingBar`, and `TendencySlider`, but most screen structure still lives inline in feature components. Tables, page headers, async loading states, error states, buttons, compact row layouts, and status messages are repeated or hand-built per screen.

Current UX friction points:

- Roster and Standings are readable, but not yet sortable, filterable, or optimized for fast comparison.
- Schedule and News use custom list layouts instead of shared dense-list conventions.
- Hub exposes several simulation actions with similar visual weight and limited preview of what each will do.
- Tactics has save feedback, but the latest QA pass found dirty-state uncertainty under keyboard-driven slider interaction.
- Match Replay has a clear basic flow, but event navigation is step-based and the report sidebar rerenders with the current event.
- Scouting is not yet a React screen, which makes it the most important V5 readiness gap.

### WHERE WE ARE GOING

V5 should treat the web app as the primary professional dashboard surface. The next UI goal is not more decoration; it is better operational density, stronger hierarchy, and reusable screen primitives that make future Scouting, Recruitment, league history, and player-detail screens cheaper to build.

V5 UI goals:

- Establish shared dashboard primitives before adding more feature screens.
- Make dense tables easier to scan through alignment, sticky identity columns, compact badges, and consistent empty states.
- Keep Hub, Tactics, League, Scouting, and Match Day visually unified enough to feel like one product.
- Split high-frequency Match Day UI into smaller memo-friendly components.
- Preserve browser-first verification and keep the layout ready for a future Electron or browser-shell package.
- Build Scouting and Recruitment on reusable table/detail patterns, not one-off screen code.

## Component Debt

- **Data tables:** Roster and Standings duplicate table shell, header, row, border, hover, and numeric-alignment decisions. Future Scouting and Recruitment screens will amplify this unless a shared `DataTable` or `DenseTable` primitive exists.
- **Async states:** Roster, Tactics, Hub, Standings, Schedule, and News each hand-roll fetch, loading, and error handling. A `useApiResource` hook plus shared `LoadingState`, `ErrorState`, and `EmptyState` components would reduce screen boilerplate and make error presentation consistent.
- **Page headers:** Hub, Roster, Tactics, Standings, Schedule, News, and Match Replay repeat heading rows with different max widths, border treatment, and action placement. A `PageHeader` with optional actions and stat chips would make new screens faster and more consistent.
- **Buttons:** Simulation, replay, save, and navigation buttons repeat long class strings and visual priority rules. Introduce an `ActionButton` with variants such as `primary`, `accent`, `secondary`, and `danger`.
- **List rows:** Schedule, News, Match Report, and Top Performers all use compact row/list ideas, but each implements layout separately. Add `CompactList`, `KeyValueRow`, and `ScoreRow` style primitives.
- **Status chips and badges:** `StatChip` exists, but starter badges, state labels, schedule status, and news tags are not unified. Create a `Badge` or `StatusPill` primitive for compact labels.
- **Rating display:** `RatingBar` works for the current roster, but the Roster table repeats five wide rating columns. V5 Scouting will need a denser rating/uncertainty pattern that can show known, fuzzy, and verified values without exploding table width.
- **Navigation:** The app uses URL search params for tabs, which is acceptable for V4. If V5 adds deeper player, scouting, or recruitment detail screens, navigation should graduate to a small route model before tab state becomes overloaded.

## Friction Audit

- **Roster:** The table has useful columns, including OVR and Potential, but there is no sorting by OVR, Potential, age, or starter status. Rating bars are legible but wide; they force horizontal scanning and make comparison slower than a compact stat matrix would.
- **Roster:** The Starter badge is helpful, but default lineup membership is the only roster status surfaced. Future roster work should add compact status and role conventions before adding player-detail links.
- **Scouting:** There is no React scouting screen yet. That blocks V5 from expressing the existing `UNKNOWN -> GLIMPSED -> KNOWN -> VERIFIED` progression in the web product and leaves no reusable uncertainty/table pattern.
- **Match Day:** Match Replay supports Back, Key Play, and Next, but it does not provide a timeline, event list, keyboard shortcuts, team/player filters, or a way to distinguish important events beyond the first key play.
- **Match Day:** Probabilities and rolls are shown as raw debug-like text. Useful for development, but this should become a collapsible details area so the main replay stays player-readable.
- **Hub:** `Play Next Match`, `Sim Week`, `Sim To User Match`, `Sim 2 Weeks`, and `Sim To Milestone` are useful, but their hierarchy and consequences are too similar. The player should be able to see which action advances one match, which advances time aggressively, and which may skip reports.
- **Hub:** The latest web QA found a stale completed-week pre-match state where the Hub enabled play while no match was available. That is primarily a state-recovery issue, but the UI also needs stronger unavailable-state messaging when the backend reports no eligible match.
- **Tactics:** Save feedback exists, but QA found that browser-driven keyboard changes to range inputs did not expose `Save Tactics`. This needs a manual accessibility check and, if confirmed, a slider behavior fix or stable automation hooks.
- **Tactics:** Eight sliders are clear but repetitive. Grouping by tactical intent, adding compact current-value summaries, and keeping dirty-state feedback near the action would reduce scanning cost.
- **League screens:** Standings, Schedule, and News share a dashboard tone, but their layouts are separate one-offs. Standings is tabular, Schedule is a custom grid list, and News is a plain feed. Shared dense-list conventions would make the league area feel more deliberate.
- **Layout consistency:** Hub uses `max-w-4xl`, Tactics uses `max-w-3xl`, and League screens use unconstrained table/list widths inside the main container. The difference is not yet harmful, but V5 should define screen-width rules by task type instead of per component.

## Rendering Performance

- **Hub state concentration:** `Hub.tsx` owns status loading, simulation state, acknowledging state, simulation results, replay loading, errors, and Match Replay switching. Each simulation or acknowledge flow can trigger several state updates and rerender the whole Hub boundary.
- **Hub callbacks:** `loadReplay`, `refreshStatus`, `handleSimulate`, and `handleAcknowledge` are recreated on each render. This is acceptable at V4 scale, but once child components are split and memoized, these should become stable callbacks or move into a small match-flow hook.
- **MatchReplay rerender scope:** `MatchReplay.tsx` keeps `eventIndex` at the top of the component, so every Back/Next click rerenders the header, stat chips, replay card, report sidebar, top performers, and acknowledge button. Split into memo-friendly sections before adding richer event navigation.
- **Replay formatting:** `valueLine()` recomputes probability and roll display strings every render for the current event. That is small today, but expanded replay data should memoize formatted debug rows or isolate them in a memoized details component.
- **Table row boundaries:** Roster and Standings render all rows inline. Current data size is small, but future Scouting and Recruitment tables may be larger and interactive. Shared row components with stable keys, memoized derived values, and optional virtualization hooks would keep expansion safe.
- **Derived roster values:** Roster already memoizes default lineup membership with a `Set`, which is good. `overallRating(player)` still runs inline for every row render; this is not a current bottleneck, but a shared table model could precompute derived columns once per response.
- **Repeated async fetch logic:** Multiple screens fetch independently on mount and do not share caching or request lifecycle logic. This can create redundant network work when users tab around and makes future polling or refresh behavior harder to reason about.
- **No profiler evidence yet:** These are structural risks identified from the React tree, not measured regressions. Before deep optimization, use React DevTools Profiler or lightweight render counters around Hub and Match Replay.

## Pre-V5 Polish Tasks

- **Design system seed:** Add shared `PageHeader`, `ActionButton`, `DataTable`, `CompactList`, `StatusMessage`, `Badge`, and `KeyValueRow` primitives before building Scouting or Recruitment screens.
- **API resource hook:** Add a small `useApiResource` hook for GET screens and a mutation helper pattern for POST actions so loading, error, stale data, and retry behavior are consistent.
- **Dense table conventions:** Define table defaults for sticky identity columns, right-aligned numeric values, centered short counts, compact badges, hover states, empty states, and optional sort affordances.
- **Scouting readiness:** Design the Scouting table around prospect identity, scouting state, uncertainty, OVR/potential projections, and action status. Reuse the same table/detail primitives planned for Roster and Recruitment.
- **Match Day split:** Break Match Replay into `ReplayHeader`, `ReplayEventPanel`, `ReplayControls`, `ReplayDebugDetails`, `MatchReportPanel`, and `TopPerformersList` before adding timeline or richer controls.
- **Hub action hierarchy:** Group simulation actions by risk and time advance. Make the primary action clear, move bulk sim actions into a secondary group, and show expected stop conditions when possible.
- **Tactics accessibility pass:** Manually verify range-input keyboard behavior, dirty-state activation, focus styling, and screen-reader labels. Add stable automation hooks if browser QA cannot reliably mutate sliders.
- **Layout rules:** Document screen-width defaults for operational screens: Hub as a two-column workspace, Tactics as a form surface, League/Scouting as full-width dense data, and Match Day as a replay cockpit.
- **Responsive QA:** Add a mobile-width visual pass to the web QA loop before V5 screens multiply. The V4 desktop pass was healthy, but mobile inspection was not automated.
- **Performance baseline:** Capture a profiler baseline for Match Replay event stepping and Hub simulation flows after the component split, then optimize only the hot paths that show up in evidence.
