# Brief 4.1 — Class Report (Offseason Signing Day Summary)

**Phase 8 Claude Design handoff. Logic settled in Phases 1–7.**

---

## 1. Teardown

**Component:** `SigningDay` in `frontend/src/components/ceremonies/Ceremonies.tsx`
**Route:** `OffseasonBeat` with `key === 'recruitment'` and `can_recruit === false` (the read-only, post-pick view).

**What exists today:**

The screen has two rendering modes:

- **Card-grid mode** (when `beat.payload.signings` is present): a tab-bar (`My / Rival / Surprise`) filters a list of `SigningCardView` cards sorted by OVR. A legend reads `N/N of your slots used · sorted by OVR`. Visually workable but the tabs have no count indicators that survive filtering, no grouping by tier, and the tab labels (`my`, `rival`, `surprise`) expose internal enum vocabulary.
- **Legacy fallback mode** (older saves without card data): a single centered hero card (`YOUR CLASS` → player name → detail string) followed by flat `other_signings` rows. This is the text-blob the synthesis describes as "Class report" — currently just a centered string with no hierarchy.

**What is silent / redundant today:**

- The heading eyebrow says "Signing Day" in the `CeremonyShell`, but this is the *results* screen (picks already made). The headline does not distinguish "here is what happened" from "here is what you still need to do."
- The three filter tabs label internal enum keys (`my`, `rival`, `surprise`) — they should read in plain language.
- The `signed_count / signing_limit` slot usage counter appears as small gray footer copy, not near the decision-relevant player signing context.
- No visual separation between "your signing(s)" and the broader league class — the tab does this work but the page has no orientation headline stating both facts at once ("You signed 2. 14 others joined the league").
- Card-grid counts per tab show parenthetically inside the tab button but don't survive when the filtered set is empty (the empty-state message is there, but the signal that caused empty — e.g. "you scouted no rivals" — is not).
- No access to a prospect's `fit_score` or `pipeline_tier` at the class-report stage — correct, those belong to recruiting, not results. The card carries `ovr`, `role`, `club_id`, `club_name`, `outcome_kind`, and `reason`. Note: `age` and `archetype` are not on `SigningCard` — they live on `RecruitmentProspectChoice` (the picker payload), not the results card.

**Phase 1–7 changes relevant here:** Recruiting can sign for full rosters (Phase 6-adjacent fix). The in-season board now targets the same class the offseason signs (Phase 0/D1). These make the payload more reliable — the player's signed prospect is now the same one they courted.

---

## 2. Information Hierarchy

**Primary:** The player's own signing outcome — who they got (or that they didn't sign), the prospect's OVR, age, archetype, and the reason they committed.

**Secondary:** League-wide class summary — how many rookies entered the league, the top OVR of the class, how many rivals signed who. The tab filter still belongs here (not primary — viewing rivals is context, not the first question).

**Cut (do not render, or demote to a collapsed/footnote state):**
- The raw `outcome_kind` enum value (already replaced by `reason` prose + the `My / Rival / Surprise` distinction).
- The `round_number` field on `SigningCard` (implementation detail; not meaningful to the player).
- The per-card `club_id` and `player_id` identifiers (backing data only).

---

## 3. Data Inventory

**Beat base fields available at the component boundary** (from `OffseasonBeatBase`):
- `beat.payload.player_signing`: `{ name, ovr, age?, role?, club_name? }` or `null`
- `beat.payload.other_signings`: `OffseasonSigning[]` — `{ name, ovr, age?, role?, club_name? }`
- `beat.payload.signings`: `SigningCard[]` — `{ player_id, name, ovr, role, club_id, club_name, user_interaction: { scouted, contacted, visited, locked_out }, outcome_kind: 'my_signing' | 'rival_signing' | 'surprise', reason, round_number }`
- `beat.payload.signed_count`: integer — how many of the player's slots are used
- `beat.payload.signing_limit`: integer — total slots this offseason
- `beat.payload.remaining_signings`: integer — slots still available
- `beat.payload.roster_size`: integer — current roster size after signings
- `beat.payload.roster_limit`: integer — hard roster cap (12)
- `beat.body`: string — fallback prose sentence

**Derived at render time:**
- Count of `my_signing` entries, `rival_signing` entries, `surprise` entries in `signings`
- Class size = `signings.length`
- Top OVR of class = `Math.max(...signings.map(s => s.ovr))`

**Fields intentionally absent** (do not invent): no per-card `potential_ceiling` (that is a Player Card field, not a SigningCard field), no `archetype_display_name` translation at this boundary (SigningCard carries `role`, not `archetype`).

---

## 4. Constraints

- Mobile viewport 390×844, no horizontal overflow. Cards in card-grid mode must stack to a single column at 390px.
- Semantic markup: the filter row should use `role="tablist"` / `role="tab"` / `aria-selected` (already present — preserve). The class summary should be a `<section>` with a visible heading.
- No new dependency. No new API endpoint — all data is on the beat payload that already loads.
- No routing change. This is a beat in the offseason ceremony flow.
- "Explain, don't decide" — show what the engine produced (who signed, why); do not editorialize or invent reasons not in `reason`.

---

## 5. Success Criteria

1. A player can immediately answer "Did I get my target?" from the primary element without scrolling.
2. The league-wide class size and the count of rival signings (the two most motivating comparison points) are visible without switching tabs.
3. The tab filter labels are in plain English: "Your Picks", "Rival Picks", "Surprises" (not the enum keys `my`, `rival`, `surprise`).
4. The legacy fallback path (no `signings` array) renders the same primary/secondary hierarchy using `player_signing` + `other_signings`, not a centered text blob.
5. No horizontal overflow at 390px width.
6. `signed_count / signing_limit` slot counter is co-located with the player's own signing, not only in small footer copy.
