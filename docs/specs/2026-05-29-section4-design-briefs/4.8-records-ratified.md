**The logic work for this screen is done. Phase 7 (D5) added the My Club / League scope filter, the `is_my_club` and `holder_club_id` fields, both empty-states, and trimmed low-impact record types. This brief is the visual restyle only.**

## 1. Teardown

**Component:** `RecordsRatified` in `frontend/src/components/ceremonies/StructuredOffseasonBeats.tsx`
**Backing type:** `RecordsRatifiedBeatPayload` in `frontend/src/types.ts`
**Beat key:** `'records_ratified'`

**What exists today (post-Phase 7):**

The screen is rendered within a `BeatShell` wrapper (which provides the `PageHeader` eyebrow + beat progress dots + "Continue" action bar). Inside `BeatShell`:

1. A header row: "Records Ratified" kicker + a My Club / League scope toggle (two small `role="group"` buttons with `aria-pressed`). The scope toggle is hidden when `records_book_empty === true`.
2. A record list: when the filtered `records` array is non-empty, a CSS grid of record cards. Each card has:
   - An orange left border (3px solid).
   - A kicker showing `titleize(record.record_type)` (e.g., "Career Eliminations").
   - The holder's name in bold.
   - A mono-font progress line: `{previous_value} -> {new_value}` in green.
   - An optional `detail` string in muted text.
   - A `<details>` element ("View evidence ‚åÑ") that expands to show `formatProofSource(record.proof_source)`.
3. Two honest empty-states (post-Phase 7):
   - `records_book_empty === true`: "The record book is empty ‚Äî records will be set as seasons are played."
   - My Club scope with no club records held: "Your club holds no league records yet."
   - Default: "No new records were set this season."

**What is still visually underdesigned:**

- The scope toggle (My Club / League) is rendered as two small flat buttons in the top-right of the header row. It blends into the kicker area and can be overlooked . The toggle is a primary navigation choice for this screen ‚Äî it should be more visually prominent.
- The record cards are visually uniform: every card has the same orange left border and the same text hierarchy. There is no visual differentiation between a record being **broken for the first time** vs. **extended to a new high** by the same holder, nor between a player club's record vs. a rival's. Only the `is_my_club` flag is available to do this differentiation.
- The `previous_value -> new_value` display in mono-font is correct but does not communicate *magnitude*. A record improved from 42 to 43 looks identical to one improved from 0 to 91.
- The "View evidence" `<details>` element consistently shows a technical source string (e.g., "Record Career Eliminations 2026"). This is proof-backed and should remain, but the string formatting could be more legible.
- The `BeatShell` progress dots and beat title ("Records Ratified") are rendered above all content by `PageHeader`. The beat is already well-titled; the main visual gap is in the record list itself.
- When the scope is "My Club" and the player holds records, this is a moment of genuine pride (their players hold league records). The current flat card grid does not differentiate or celebrate this.
- The default empty-state ("No new records were set this season.") appears even when the player could switch to League scope and see records there ‚Äî the scope toggle being in the header may not be noticed. Consider whether the empty-state should prompt a scope switch.

**Phase 7 changes that are settled and must not be redesigned:**
- **`is_my_club` and `holder_club_id` fields** are now on every `RatifiedRecordEntry`. Use them for visual differentiation ‚Äî do not re-derive club membership from name-matching.
- **`records_book_empty` field** distinguishes "no records at all" from "no new records this season." Both empty-states must remain honest and distinct.
- **Default scope:** Opens on "My Club" when `hasMyClubRecords === true`, else falls back to "League." This logic is in the component (`useState` initializer) ‚Äî preserve it.
- **Record types retained** (post-Phase 7 trim): `career_eliminations`, `career_catches`, `most_championships`, and all team records. The trimmed types (`career_dodges`, `most_seasons_at_one_club`) are gone ‚Äî do not re-introduce them.

## 2. Information Hierarchy

**Primary:** Whether the player's club holds a record. The `is_my_club` flag on each `RatifiedRecordEntry` makes this trivial to surface. A My Club record card should read differently from a League-only record card.

**Secondary:** The record milestone itself ‚Äî holder name, record type, and the new value. The `previous_value -> new_value` progression communicates that a record was beaten, not just set.

**Tertiary:** Proof and context ‚Äî the `detail` string (contextual note, e.g., season stats) and the `<details>` evidence expansion.

**Cut or demote:**
- The "View evidence" technical source string should remain accessible but remain collapsed by default ‚Äî it is proof infrastructure, not primary copy. The current `<details>` is correct; ensure the closed state does not look broken or like a call-to-action.
- The scope toggle buttons' current small-text design ‚Äî elevate them so the toggle is not overlooked .

## 3. Data Inventory

**From `RecordsRatifiedBeatPayload`:**
- `records`: `RatifiedRecordEntry[]` ‚Äî may be empty
  - `record_id?`: string ‚Äî optional unique identifier
  - `record_type`: string ‚Äî e.g., `'career_eliminations'` (post-Phase 7 only the retained types appear)
  - `holder_name`: string ‚Äî player or club name
  - `previous_value`: number ‚Äî the prior record value
  - `new_value`: number ‚Äî the new record value
  - `detail`: string ‚Äî contextual description (e.g., "42 eliminations across 2 seasons")
  - `proof_source?`: string ‚Äî backend proof identifier
  - `holder_club_id?`: string ‚Äî the club ID of the record holder (Phase 7 field)
  - `is_my_club?`: boolean ‚Äî true when the holder is the player's club (Phase 7 field)
- `records_book_empty?`: boolean ‚Äî true when no records exist league-wide (fresh league)

**Component state:**
- `scope`: `'my_club' | 'league'` ‚Äî drives filtering. Initialized to `'my_club'` when `hasMyClubRecords`, else `'league'`.
- `allRecords`: `beat.payload.records ?? []`
- `records` (filtered): `scope === 'my_club' ? allRecords.filter(r => r.is_my_club) : allRecords`

**Derivable at render time:**
- `hasMyClubRecords`: `allRecords.some(r => r.is_my_club)`
- `delta`: `record.new_value - record.previous_value` ‚Äî always positive (a record is only ratified when broken upward)
- Display name for `record_type`: `titleize(record.record_type)` already converts underscores to title case

**Fields intentionally absent (do not invent):** No season-by-season breakdown, no per-player career arc, no rival comparisons beyond the league-scope view. This beat is a snapshot, not a history.

## 4. Constraints

- Desktop-first matrix (1440x900 / 1366x768 / 1280x720). Mobile layouts are explicitly a non-goal.
- Semantic markup: the scope toggle must use `role="group"` with `aria-label="Records scope filter"` (already implemented ‚Äî preserve). Each button must have `aria-pressed` (already implemented ‚Äî preserve).
- The `<details>` / `<summary>` proof expansion must remain accessible and keyboard-operable (native HTML ‚Äî no change needed, just preserve).
- No new dependency, no new API call. All data is on `beat.payload`.
- No routing change. This is a sequential offseason beat.
- The "Continue" CTA in `BeatShell` must remain reachable at 390px without requiring the user to scroll past a long record list.
- "Explain, don't decide" ‚Äî the record values and proof source are engine-produced facts. Do not editorialize. The `detail` string is server-authored.

---

## 5. Success Criteria

1. The My Club / League scope toggle is visually prominent enough that a player sees it without searching the header area.
2. Records where `is_my_club === true` are visually distinguished from rival records ‚Äî a player should be able to identify "my club's records" at a scan.
3. Both empty-states render their distinct messages correctly and are not visually identical to each other.
4. When the "My Club" scope is empty but League records exist, the empty-state offers a clear path to switch scope (e.g., "Switch to League view to see all records" or an auto-prompt).
5. The proof `<details>` expander is clearly a "drill down" affordance, not a primary CTA ‚Äî its collapsed state reads as complete.
6. No horizontal overflow at 1280x720 minimum desktop width.
7. The trimmed record types (`career_dodges`, `most_seasons_at_one_club`) do not appear ‚Äî confirmed by the Phase 7 backend trim, not the frontend (no frontend filter needed; do not add one).
